Guide for building a Kakao Business CS agent

카카오 상담을 안전하게 읽고, 분류하고, 답장 초안을 만드는 오케스트레이션

이 문서는 지인이 같은 구조를 만들 수 있도록 정리한 구현 가이드입니다. 카카오 비즈 상담 목록을 주기적으로 확인하고, 새 문의를 분류하고, 답장 초안 또는 승인된 답변을 만들고, 사람이 봐야 하는 케이스를 넘기는 방법을 설명합니다.

문서 범위

  • 포함: 로그인 설정 위치, 상담 스캔, 폴링 루프, 답장 정책, 중복 방지, 승인 규칙
  • 제외: 이지어드민에서 주문을 조회하는 상세 절차
  • 원칙: 비밀번호와 고객 원문은 코드, 프롬프트, 로그에 저장하지 않음
  • 업데이트: 2026-07-03 KST

1. 전체 구조

이 오케스트레이션은 하나의 거대한 자동응답 봇이 아니라, 작은 역할을 가진 단계들을 순서대로 묶은 구조입니다. 핵심은 상담 목록을 읽는 단계와 고객에게 보내는 단계 사이에 분류, 안전 규칙, 사람 승인을 넣는 것입니다.

1
세션 확인카카오 비즈 상담 화면에 로그인되어 있는지 확인합니다. 로그인, 2차 인증, 본인확인 화면이면 자동화를 멈춥니다.
2
상담 목록 스캔읽지 않은 상담, 새 메시지, 최근 변경된 상담만 가져옵니다. 광고, 채널추가, 시스템 자동 메시지는 제외합니다.
3
히스토리 분석오늘 타임라인, 담당자 응대 여부, 고객 추가 메시지, 이미지 수신 여부, 강성 신호를 계산합니다.
4
문의 유형 분류단순 문의, 배송 관련 문의, 파손/누락/오배송, 환불/취소, 식품안전, 강성 고객, 판단불가로 나눕니다.
5
답장 초안 또는 응대자동 발송을 기본값으로 두지 말고, 처음에는 초안 모드로 운영합니다. 발송은 현재 화면과 메시지 내용을 확인한 뒤 실행합니다.
6
기록과 에스컬레이션사람 확인이 필요한 건은 내부 기록과 알림으로 넘기고, 고객에게 임의 결론을 말하지 않습니다.
주문 조회나 배송 사실 확인이 필요한 경우에는 내부 주문 시스템 또는 담당자 확인 단계로 넘기면 됩니다. 이 문서는 그 내부 시스템을 어떻게 조회하는지는 다루지 않습니다.

2. 로그인과 환경설정

로그인 정보는 코드 안에 직접 적지 않습니다. 개발 초기에는 .env 파일, 운영 단계에서는 OS Keychain, 1Password, AWS Secrets Manager 같은 안전한 저장소를 권장합니다. 프롬프트, HTML 문서, 로그, 데이터베이스에는 비밀번호를 남기지 않습니다.

필수 설정값

키 예시 용도 주의
KAKAO_BIZ_URL https://business.kakao.com/{channel_id}/chats 상담 목록 진입 URL 채널 ID가 맞는지 사람이 확인
KAKAO_LOGIN_ID your-login-id 로그인 계정 로그에는 출력 금지
KAKAO_LOGIN_PASSWORD secure-placeholder 초기 로그인 보조 가능하면 브라우저 저장 세션과 Keychain 사용
CS_MODE draft 또는 assist 초안만 만들지, 제한적으로 발송할지 결정 처음 1-2주는 draft 권장
POLL_ACTIVE_SECONDS 180 운영시간 중 상담 확인 간격 너무 짧으면 계정/브라우저에 부담
POLL_IDLE_SECONDS 600 비운영시간 확인 간격 야간 자동 발송은 제한
INTERNAL_ALERT_WEBHOOK https://hooks.example/... 담당자 확인 필요 알림 Webhook URL은 비밀값으로 취급
.env.example
KAKAO_BIZ_URL=https://business.kakao.com/YOUR_CHANNEL_ID/chats
KAKAO_LOGIN_ID=your-login-id
KAKAO_LOGIN_PASSWORD=replace-with-secure-store-value

CS_MODE=draft
POLL_ACTIVE_SECONDS=180
POLL_IDLE_SECONDS=600
IMMEDIATE_WATCH_SECONDS=120
IMMEDIATE_WATCH_INTERVAL_SECONDS=20

INTERNAL_ALERT_WEBHOOK=replace-with-secure-store-value
STATE_DB_PATH=./data/kakao-cs-state.sqlite3
RUN_LOG_DIR=./runs
2차 인증, CAPTCHA, 본인확인, 비밀번호 재입력 화면이 보이면 자동화가 직접 처리하면 안 됩니다. 이 경우 상태를 blocked_login_required로 기록하고 사람이 브라우저에서 세션을 복구해야 합니다.

3. 상담 규칙 설계

좋은 CS 오케스트레이션은 답변 문장을 잘 쓰는 것보다 먼저 멈춰야 할 상황을 정확히 압니다. 아래 규칙은 최소 기준입니다.

자동 응대 가능

상품 보관법 조리법 영양 정보 배송 정보 수집

사실 확인이 단순하거나, 고객에게 필요한 정보를 요청하는 단계입니다.

초안만 권장

환불 취소 파손 오배송

내부 확인이 필요하거나 고객 불만으로 이어질 수 있는 단계입니다.

고객 답변 중지

식품안전 강성 고객 담당자 응대 중

바로 담당자 알림으로 넘기고 AI가 추가 답변하지 않습니다.

항상 제외

광고 채널추가 시스템 메시지

고객 문의로 분류하지 않고 처리 대상에서 제외합니다.

규칙 파일 예시

rules.yaml
mode: draft
timezone: Asia/Seoul
business_hours:
  weekday_start: "10:00"
  weekday_end: "16:00"

ignore_message_types:
  - channel_added
  - auto_reply
  - advertisement
  - system_notice

hard_stop_rules:
  food_safety:
    keywords: ["배탈", "설사", "구토", "상함", "곰팡이", "이물질", "벌레", "식중독"]
    customer_reply: false
    internal_alert: true
  angry_customer:
    keywords: ["책임자", "사장", "본사", "고소", "신고"]
    profanity_or_threat: true
    customer_reply: "one_empathy_message_only"
    internal_alert: true
  manager_active:
    customer_reply: false
    internal_alert: true

info_required:
  simple_question: ["name_optional"]
  delivery_or_order_related: ["name", "phone", "message_summary"]
  damaged_or_wrong_item: ["name", "phone", "photo_or_description"]

send_policy:
  default: draft_only
  allow_auto_send_for:
    - information_collection
    - simple_product_info
  require_human_approval_for:
    - refund
    - cancellation
    - damage
    - wrong_delivery
    - food_safety
    - angry_customer
    - any_order_conclusion
고객에게 결론을 말하기 전에 내부 사실 확인이 필요한 유형이 있습니다. 주문, 환불, 취소, 보상, 재출고, 식품안전은 자동 결론을 내리지 않는 편이 안전합니다.

4. 확인 루프와 폴링 기준

상담 확인은 무조건 빠를수록 좋은 것이 아닙니다. 계정 안정성, 담당자 개입 가능성, 중복 답변 방지를 함께 고려해야 합니다.

권장 간격

상황 간격 이유
평일 운영시간 3분 응답 속도와 브라우저 부담의 균형
주말 또는 비운영시간 5-10분 단순 정보는 대응하되 민감 건은 담당자 확인으로 넘김
방금 답장한 직후 20초 간격, 최대 2분 고객이 바로 이름/연락처를 답할 수 있으므로 짧게 추가 감시
로그인 오류 후 자동 재시도 금지 2차 인증과 보안 화면은 사람이 복구

폴링 루프 의사코드

polling-loop.pseudo
while true:
  run_id = create_run(mode="read_or_preview")

  session = check_kakao_session()
  if session.status in ["login_required", "captcha", "two_factor_required"]:
    record_blocker(run_id, session.status)
    notify_internal("카카오 로그인 복구 필요")
    sleep(POLL_IDLE_SECONDS)
    continue

  conversations = scan_unread_conversations()
  candidates = filter_out_system_messages(conversations)

  for chat in candidates:
    if already_processed(chat.last_message_key):
      continue

    history = read_recent_history(chat.id, days=3)
    state = analyze_history(history)

    decision = classify_case(history, state, rules)
    draft = build_reply_or_handoff(decision, history)

    if decision.action == "stop_customer_reply":
      record_internal_case(chat, decision, draft)
      mark_processed(chat.last_message_key)
      continue

    if CS_MODE == "draft":
      save_draft(chat, draft)
      notify_internal_if_needed(chat, decision)
    else:
      send_result = send_with_preflight(chat, draft)
      verify_sent_message(chat, send_result)

    mark_processed(chat.last_message_key)

  if any_reply_sent_or_draft_created:
    watch_immediate_replies(max_seconds=120, every_seconds=20)

  sleep(next_poll_interval())

루프는 반드시 last_message_key 같은 중복 방지 키를 가져야 합니다. 같은 메시지를 여러 번 처리하면 고객에게 중복 답변이 나가거나 내부 알림이 반복됩니다.

5. 답장 처리 방법

답장은 크게 세 종류로 나눕니다. 첫째, 고객에게 추가 정보를 요청하는 답장. 둘째, 단순 정보 안내. 셋째, 담당자에게 넘기는 내부 메시지입니다.

발송 전 체크

현재 열린 상담방 URL 또는 chat id가 답장 대상과 일치하는지 확인합니다.
담당자가 오늘 이미 응대한 흔적이 있으면 고객 답변을 멈춥니다.
식품안전, 강성 고객, 환불/취소 결론, 보상 약속이 포함되어 있으면 사람 승인을 받습니다.
고객명, 연락처, 주문정보 같은 개인정보를 다른 상담과 섞지 않았는지 확인합니다.
깨진 문자, 제어 문자, 과한 이모지, 확정되지 않은 날짜 표현을 제거합니다.

정보 요청 답장 템플릿

reply-template.txt
안녕하세요 고객님.
확인을 도와드리기 위해 아래 정보를 남겨주세요.

1. 성함
2. 연락처
3. 문의 내용

확인 후 안내드리겠습니다.

다른 문의사항 있으시면 편하게 말씀해 주세요 :)

사람 확인 필요 내부 알림 템플릿

internal-handoff.txt
[카카오 상담 확인 필요]
상담방: {chat_url_or_chat_id}
문의유형: {case_type}
고객정보 수집: {collected_fields}
AI 판단: {summary}
차단 사유: {food_safety | angry_customer | manager_active | needs_order_confirmation | needs_human_decision}
권장 다음 조치: {safe_next_action}

주의: 고객 원문 전체와 민감정보는 내부 기록 규칙에 따라 최소화합니다.
처음 구축할 때는 모든 고객-facing 메시지를 draft_only로 두는 것이 좋습니다. 1-2주 동안 사람이 검수하면서 어떤 유형이 안전한지 확인한 뒤, 단순 정보 안내부터 자동 발송을 열어갑니다.

6. 상태 저장과 중복 방지

오케스트레이션은 기억력이 있어야 합니다. 다만 고객 원문 전체, 비밀번호, 세션 토큰은 저장하지 않습니다. 저장해야 하는 것은 처리 상태, 최소 요약, 중복 방지 키, 차단 사유입니다.

state-schema.sql
create table if not exists cs_runs (
  id text primary key,
  started_at text not null,
  finished_at text,
  mode text not null,
  status text not null,
  blocker text,
  evidence_summary text
);

create table if not exists cs_items (
  id text primary key,
  run_id text not null,
  chat_id text not null,
  last_message_key text not null,
  case_type text not null,
  action text not null,
  status text not null,
  collected_fields text,
  safe_summary text,
  next_action text,
  created_at text not null,
  unique(chat_id, last_message_key)
);

create table if not exists cs_drafts (
  id text primary key,
  item_id text not null,
  draft_text text not null,
  approval_status text not null default 'pending',
  sent_at text,
  verification text
);

상태값 기준

상태 뜻 다음 행동
completed_draft 초안 생성 완료, 발송은 하지 않음 사람 검수
completed_sent 승인된 답변 발송 후 화면 검증 완료 다음 폴링 대기
blocked_login_required 로그인/2차 인증/본인확인 필요 사람이 브라우저 세션 복구
blocked_human_required 민감 케이스라 AI 답변 중지 담당자 확인
rework_required 초안 또는 기록이 검증 기준을 통과하지 못함 규칙 또는 프롬프트 수정

7. 검증과 운영 게이트

오케스트레이션은 “돌아갔다”가 아니라 “증거가 있다”로 끝나야 합니다. 특히 고객 답변, 내부 알림, 상담 종료 같은 작업은 실행 전 승인과 실행 후 검증이 필요합니다.

실행 전 계약

완료 판정

상담 화면과 채널이 현재 실행에서 확인되었습니다.
각 상담마다 분류, 차단 여부, 다음 행동이 기록되었습니다.
고객에게 실제 발송한 경우, 발송 후 화면에서 메시지 존재를 확인했습니다.
초안 모드인 경우 not_executed_yet가 명확히 기록되었습니다.
막힌 경우 blocked 또는 rework_required로 끝냈고 성공으로 보고하지 않았습니다.

8. 구축 체크리스트

카카오 비즈 상담 URL과 채널 ID를 확인했다.
로그인 정보는 .env 또는 비밀 저장소에만 넣고 코드에 하드코딩하지 않았다.
처음 운영 모드는 draft_only로 설정했다.
식품안전, 강성 고객, 담당자 응대 중 케이스는 자동 답변 금지로 설정했다.
폴링 간격, 즉시 응답 감시 시간, 재시도 제한을 정했다.
상담별 중복 방지 키를 저장한다.
고객 원문 전체와 개인정보를 장기 저장하지 않는다.
